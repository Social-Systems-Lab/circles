# Onboarding Architecture — 30s Flow (Updated)

## Goal

Get users from signup → contribution as fast as possible.

Primary objective:

**Minimize friction, maximize activation**

---

## Core Principle

Onboarding is not data collection.

It is the **first step of the Contribution Engine**.

---

## Target Flow (30–60 seconds)

1. Welcome
2. Terms
3. Identity (member / builder)
4. Skills selection
5. Mission / interests
6. MCP (contribution intent)
7. Profile basics
8. Enter platform

---

## Step Breakdown

### 1. Welcome

- Clear value proposition
- “This is where you contribute”

---

### 2. Terms

- Fast accept
- No friction

---

### 3. Identity

Question:

Are you:

- Contributing as an individual
- Working on a project / initiative

Purpose:

- segment users early
- enable future flows

---

### 4. Skills Selection

Core input:

- what can you offer?

Used for:

- matching
- task suggestions
- contribution activation

---

### 5. Mission / Interests

What do you care about?

Used for:

- relevance
- filtering opportunities

---

### 6. MCP (Contribution Intent)

Key new step.

Capture:

- monthly contribution willingness (amount)
- volunteering willingness

Stored as:

donationIntent

Purpose:

- measure economic potential
- guide future activation
- inform funding strategy

Important:

This is **intent only**, not commitment.

---

### 7. Profile Basics

Minimal:

- name
- optional image
- optional bio

Everything else later.

---

### 8. Enter Platform

User lands in:

- feed / explore / tasks (future)

Immediate goal:

→ show ways to contribute

---

## Completion Strategy

Use a **progress bar**, but:

- allow early exit
- encourage completion later

Profile completeness becomes:

- soft guidance
- not a blocker

---

## Activation Definition

A user is activated when they do one of:

- offer a skill
- respond to a need
- join a project
- complete a task (future)

---

## MCP Integration

Onboarding feeds directly into:

Contribution Engine → Measurement layer

MCP becomes:

- baseline economic signal
- activation targeting input

---

## Design Principles

- fast
- clear
- purposeful
- no dead steps
- every step feeds the system

---

## What NOT to do

- over-collect data
- require unnecessary fields
- delay entry into platform
- confuse users with too many options

---

## Future Enhancements

- adaptive onboarding (based on answers)
- dynamic prompts
- contribution suggestions immediately after onboarding
- personalized feed based on skills + MCP

---

## TL;DR

Onboarding is not a form.

It is the **first interaction with the Contribution Engine**.

Goal:

Get users to meaningful contribution as quickly as possible.
