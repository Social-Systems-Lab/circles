# Kamooni System Messages Engine

Developer architecture overview for platform messaging.

---

# 1. Purpose

Kamooni uses a **system messaging engine** to communicate with users directly inside the chat interface.

This replaces email-based onboarding and platform announcements.

System messages include:

• Welcome messages when a user signs up
• Platform broadcasts sent by admins
• Announcement threads where replies are disabled

These messages appear as normal chat conversations but are generated by the platform.

---

# 2. Core concept

System messages behave like **normal chat conversations**, but:

• sender DID is a system identity
• replies may be disabled
• messages are generated by the platform

This ensures system communication uses the same UI as normal messaging.

---

# 3. System sender identity

System messages are sent from a **virtual DID identity**.

Example:

system:kamooni

Properties:

• fixed DID
• stable sender name
• stable avatar (Kamooni logo)
• cannot log in

The identity exists purely as a sender reference.

Files involved:

src/lib/chat/system-sender.ts

---

# 4. Welcome message pipeline

When a user signs up:

1. Platform ensures a welcome conversation exists
2. Conversation metadata is created
3. System message is inserted

Relevant files:

src/lib/data/mongo-chat.ts
src/lib/chat/system-message-templates.ts
src/lib/data/system-message-events.ts

Conversation metadata contains:

source: "signup"
repliesDisabled: true

Welcome messages are rendered as **markdown**.

---

# 5. Platform broadcast system

Admins can send platform broadcasts.

Admin UI:

/admin → System Messages panel

Broadcast workflow:

Save → stores draft
Preview to Self → sends test message to admin
Broadcast → activates message for all users

Relevant files:

src/lib/data/platform-broadcasts.ts
src/components/modules/admin/system-messages-tab.tsx

Broadcast messages are delivered through the **announcement conversation pipeline**.

---

# 6. Announcement conversations

Announcement conversations behave like chat rooms but:

• replies are disabled
• messages come from system sender
• UI hides reply controls

Conversation flags:

conversationType: "announcement"
repliesDisabled: true

These flags are set in:

src/lib/data/mongo-chat.ts

UI enforcement occurs in:

src/components/modules/chat/chat-room.tsx

---

# 7. Reply disabling (server + UI)

Replies are disabled in two layers.

Server enforcement:

src/components/modules/chat/mongo-actions.ts

If conversation type is announcement:

return error
"Replies are disabled for this conversation"

UI enforcement:

chat-room.tsx hides reply actions.

---

# 8. Message format

System messages support:

markdown

Admin editor includes a lightweight markdown toolbar:

• Bold
• Italic
• Link
• Bullet list

Toolbar is implemented in:

system-messages-tab.tsx

Messages are stored as:

format: "markdown"

---

# 9. Conversation normalization

During chat list rendering:

Mongo conversations are normalized into ChatRoomDisplay objects.

Normalization occurs in:

src/lib/data/mongo-chat.ts

This step sets stable flags:

conversationType
repliesDisabled

The chat UI relies on these flags.

---

# 10. Architectural advantages

Using chat-native system messaging provides:

• consistent UI
• real-time delivery
• no email dependency
• user attention inside platform

This architecture allows future features like:

• community announcements
• project notifications
• governance voting alerts
